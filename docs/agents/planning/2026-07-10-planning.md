## Today's Question

How can classical planning methods (PDDL, optimal planners) be integrated with LLMs to overcome the planning limitations of pure LLM approaches?

## Main Paper

### Metadata

- Title: LLM+P: Empowering Large Language Models with Optimal Planning Proficiency
- Authors: Bo Liu, Yuqian Jiang, Xiaohan Zhang, Qiang Liu, Shiqi Zhang, Joydeep Biswas, Peter Stone
- Year: 2023
- Venue: arXiv (2304.11477)
- Link: https://arxiv.org/abs/2304.11477

### Why this paper?

LLM+P is the first framework to systematically combine LLMs with classical planners. It directly defines the "hybrid planning" paradigm and serves as the canonical reference for the key question in the topic: should agents plan purely with LLMs or delegate planning to symbolic planners? After Day 1's survey established the landscape, this paper fills in the first concrete architecture.

### Core Problem

LLMs cannot reliably solve long-horizon planning problems — they generate plausible-sounding plans that are often infeasible, miss critical steps, or violate constraints. Classical planners, however, can efficiently find correct or even optimal plans, but require structured input (PDDL) that is inaccessible from natural language. The problem is bridging NL understanding with sound planning.

### Main Idea

LLM+P proposes a three-stage pipeline:

1. **NL → PDDL (LLM)**: An LLM translates a natural language planning problem description into a PDDL domain file + problem file.
2. **PDDL → Plan (Classical Planner)**: An off-the-shelf classical planner (e.g., Fast Downward) finds a correct or optimal plan.
3. **PDDL Plan → NL (LLM)**: A second LLM call translates the symbolic plan back into natural language.

This keeps the LLM in its zone of strength (language translation) and offloads the actual planning computation to algorithms that guarantee soundness.

### Technical Details

- The paper defines a diverse benchmark of 7 planning problem types: blocksworld, logistics, ferry, miconic, rover, satellite, and a new "grid" domain.
- Each problem type has a pre-written PDDL domain template. The LLM only needs to fill in the problem instance (objects, initial state, goal).
- For each problem, they generate 5-10 random instances covering increasing difficulty (number of objects, goal complexity).
- Metrics: plan feasibility (any valid plan found) and plan optimality (whether the found plan achieves minimum steps).
- Baselines: GPT-3.5 and GPT-4 prompted with the same NL problem descriptions using standard (zero-shot, few-shot) and chain-of-thought prompting.
- The classical planner used is Fast Downward with optimal search (A* + landmark heuristic).

### Results

- **LLM+P achieves 100% plan feasibility** across all benchmark problems.
- **Pure GPT-4 achieves < 40% plan feasibility** on most problems, and often 0% on harder instances.
- Even with chain-of-thought and few-shot prompting, LLMs fail to produce feasible plans for problems requiring more than ~5 steps.
- LLM+P also produces *optimal* plans on all problems (since the classical planner is run with optimal search settings).
- Key failure modes for pure LLMs: missing preconditions (e.g., not unloading before moving), ordering errors, and object confusion.

### Limitations

- Requires pre-written PDDL domain templates for each problem type — does not scale to arbitrary novel planning problems without manual domain engineering.
- The NL→PDDL translation step still has errors, especially for complex goal specifications. The paper reports ~15% translation error rate on the hardest problems.
- Assumes the planning problem is fully observable and deterministic — cannot handle partial observability, stochastic effects, or continuous state spaces.
- The two LLM calls (NL→PDDL, PDDL→NL) add latency and cost.
- Only tested on classical planning domains, not on real-world agent tasks with tool use.

### Modern perspective (2026)

The hybrid LLM + symbolic planning paradigm has since been extended significantly:

- **NL2Plan** (2024) replaces manual domain templates with fully automated PDDL generation from scratch, reducing the domain engineering burden.
- **SayCan** and **TAPE** integrate grounded tool/environment constraints into the planning loop, moving beyond purely symbolic domains.
- **MCTS-based hybrid approaches** (e.g., LLM-MCTS, RAP) use search instead of PDDL, trading optimality guarantees for broader applicability.
- The core insight of LLM+P — that LLMs should handle the "interface" while classical algorithms handle the "computation" — has become a mainstream design principle in agent systems.

### Research takeaway

LLM+P established that **pure LLM planning is fundamentally unreliable** for non-trivial problems and that **hybridization with symbolic methods is necessary for correctness guarantees**. This is a key map node: the planning topic is bifurcated into "pure LLM prompting" approaches and "hybrid/grounded" approaches, with LLM+P at the root of the latter.

**What it is**: A prompting strategy that explicitly separates plan generation from step-by-step execution. **Relation to main paper**: Plan-and-Solve represents the "pure LLM" approach to planning — all planning happens in the LLM's forward pass, with no external solver. Comparing it with LLM+P reveals the fundamental tradeoff: pure LLM planning is flexible (no domain engineering) but brittle (no correctness guarantees), while hybrid planning is sound but requires structured problem interfaces.

**Why it matters**: Plan-and-Solve was one of the most widely adopted prompting strategies in 2023-2024, reaching comparable performance with 8-shot CoT on math reasoning. Its limitations (missing-step errors, inability to recover from suboptimal decomposition) motivated the hybrid planning approaches that followed.

### Paper 2: TAPE — Tool-Guided Adaptive Planning

- Authors: Jongwon Jeong, Jungtaek Kim, Kangwook Lee
- Year: 2026
- Link: https://arxiv.org/abs/2602.19633

**What it is**: A modern hybrid planning framework that uses an external solver on a plan graph to find feasible paths. **Relation to main paper**: TAPE extends the LLM+P hybrid paradigm from classical PDDL to tool-augmented environments. Instead of a single PDDL planner, it plans over a graph of possible steps and uses an external solver for path-finding. The constrained execution component is a new contribution not present in LLM+P.

**Why it matters**: TAPE shows that the hybrid planning paradigm (LLM for interface + external algorithm for search) generalizes beyond classical planning to modern agent environments. It improved hard settings by 21% on average, demonstrating the ongoing relevance of LLM+P's core insight.

## Current Understanding

Planning for LLM agents bifurcates along a fundamental axis: **pure LLM planning** (prompting-based: Plan-and-Solve, ReAct, Tree-of-Thought) vs. **hybrid planning** (LLM + external solver: LLM+P, TAPE, NL2Plan).

Pure LLM planning is the most flexible approach — it works with any task describable in natural language and requires no domain engineering. But it is inherently unreliable: LLMs produce plausible-sounding but incorrect plans, especially for tasks requiring >5 steps, strict preconditions, or precise ordering.

Hybrid planning sacrifices flexibility for correctness. By converting the planning problem into a structured representation (PDDL, plan graph, state machine) solvable by classical algorithms, it guarantees soundness and often optimality. The bottleneck shifts to the interface — can the LLM reliably translate NL into the structured format?

Day 1 (survey) introduced the three-stage framework: decomposition, execution, re-planning. Day 2 adds the crucial distinction: the decomposition and execution stages can be implemented either purely within the LLM or by delegating to external solvers, with significant reliability implications.

## Key Concepts

- **Hybrid planner**: An architecture combining LLM (for NL understanding/generation) with a classical planner or solver (for sound search/optimization).
- **PDDL (Planning Domain Definition Language)**: A standardized formal language for representing planning problems (domain + instance).
- **Plan-as-translation vs. plan-as-generation**: The distinction between treating planning as NL↔structured-language translation vs. treating planning as end-to-end LLM generation.
- **Plan graph**: Aggregation of multiple candidate plans into a graph structure for external solution search (used in TAPE).
- **Constrained decoding**: Biasing LLM token generation during execution to respect feasibility constraints.
- **Adaptive re-planning**: Triggering re-planning when environmental feedback deviates from the predicted state, rather than at fixed intervals.

## Open Questions

- Does the hybrid approach scale to domains without pre-engineered PDDL templates? NL2Plan partially addresses this, but fully automatic domain compilation for arbitrary tasks remains open.
- What is the right "sweet spot" between pure LLM and hybrid planning — should every planning problem be PDDL-encoded, or only those with strict correctness requirements?
- Can constrained decoding (TAPE) replace explicit planning in some settings, or is it only a supplement?
- How should re-planning frequency be determined in adaptive systems — at fixed intervals, on failure detection, or based on model uncertainty?
- Can search-based planning (MCTS, Monte Carlo Tree Search) be effectively combined with LLMs without the full PDDL pipeline?

## Possible Thesis Ideas

- **Dynamic planner selection**: An agent that dynamically decides whether to use pure LLM planning or hybrid LLM+classical planning based on task characteristics (complexity, safety requirements, domain structure). The model would learn to estimate when soundness guarantees are worth the interface cost.
- **Unified NL↔PDDL interface**: A fine-tuned model specialized for NL-to-PDDL translation, trained on synthetic data from classical planning problems, enabling hybrid planning for arbitrary tasks without manual domain engineering.
- **Plan graph + LLM execution**: Extend TAPE's plan-graph approach to combine multiple candidate LLM-generated plans (not just one) into a graph, then use a learned verifier to select the feasible path — removing the need for an external classical planner while retaining soundness.
- **Re-planning policy as a learned skill**: Train a lightweight classifier that predicts when a plan will fail before execution, enabling anticipatory re-planning instead of reactive re-planning.
