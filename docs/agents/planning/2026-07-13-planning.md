## Today's Question

Beyond flat decomposition and PDDL-based planning, how do hierarchical structure and search-based methods make LLM agent planning more reliable?

## Main Paper

### Metadata

- Title: A Roadmap to Guide the Integration of LLMs in Hierarchical Planning
- Authors: Israel Puerta-Merino, Carlos Núñez-Molina, Pablo Mesejo, Juan Fernández-Olivares
- Year: 2025
- Venue: arXiv (Jan 2025, updated Nov 2025)
- Link: https://arxiv.org/abs/2501.08068

### Why this paper?

After establishing the general planning landscape (Day 1 survey) and the hybrid LLM+classical paradigm (Day 2 — LLM+P, TAPE), this paper opens a third dimension: **hierarchical planning (HTN)**. HTN is a well-established subfield of classical AI planning that structures tasks hierarchically through method decomposition, which aligns naturally with how LLMs think about tasks. This paper provides a structured roadmap for integration.

### Core Problem

LLM-based planning methods (Plan-and-Solve, ReAct, ToT) treat planning as flat step generation or search. Classical Hierarchical Planning (HTN) uses **task decomposition via methods** — a high-level task is broken down into subtasks by selecting appropriate methods from a domain library. The question is: can LLMs serve as the "knowledge source" for HTN domain engineering, plan generation, or both?

### Main Idea

The paper proposes a **five-level integration roadmap** for injecting LLMs into HTN planning:

| Level | Name | What the LLM does |
|-------|------|-------------------|
| 0 | Manual HTN | LLM not involved; traditional HTN |
| 1 | LLM-assisted domain engineering | LLM helps write HTN domain models (methods, tasks, preconditions) |
| 2 | LLM-based task refinement | LLM decomposes abstract tasks into method candidates |
| 3 | LLM-guided search | LLM prunes the HTN search space by ranking method applicability |
| 4 | Fully integrated LLM-HTN | LLM subsumes the planner; HTN structure purely as an interpretable scaffold |

At Level 1, the LLM acts as a domain authoring assistant — translating natural language task descriptions into structured HTN methods. At Level 2, the LLM becomes part of the planning process, generating task decompositions online. At Level 3, the LLM serves as a heuristic guide for the HTN planner's search. At Level 4, the LLM effectively replaces the planner while retaining HTN's structured output.

### Technical Details

The paper surveys several approaches mapped to this framework:

1. **LLM+HTN systems** (Level 1-2): Systems like PlanGen (Li 2023) and LLM-HTN (Puerta-Merino 2024) use LLMs to generate HTN method hierarchies from NL descriptions. Key finding: LLMs can generate syntactically valid HTN methods ~70% of the time, but correctness requires human verification.

2. **LLM as search heuristic** (Level 3): The LLM scores candidate HTN method applications, pruning the search tree. Early experiments show 5-10× reduction in HTN search space vs. blind search, with comparable plan quality.

3. **Hierarchical decomposition with LLM guidance** (Level 2): Methods like AdaPlanner (Sun 2024) and LLM-3 (Li 2025) use LLMs for hierarchical task decomposition without explicit HTN formalism, achieving better plan reliability on long-horizon tasks.

4. **Key challenges** identified:
   - **Domain acquisition bottleneck**: HTN methods require structured domain knowledge that LLMs don't naturally produce without hallucination
   - **Verification gap**: How to verify that an LLM-generated HTN method library is complete and correct
   - **Method selection at scale**: As HTN libraries grow, LLM-based method selection degrades
   - **Evaluation**: No standard benchmark for LLM+HTN integration exists

### Research Takeaway

The roadmap's key insight is that **HTN provides a middle ground** between pure LLM planning (flexible but unreliable) and classical PDDL planning (correct but rigid). HTN's hierarchical structure matches how LLMs naturally decompose tasks, while its formal method framework provides correctness constraints. The "sweet spot" is likely Level 2-3, where the LLM generates or selects methods, but the HTN planner ensures structural consistency.

### Modern Perspective

Since this roadmap was published (Jan 2025), the field has moved toward **reasoning-guided planning** (e.g., o1-style chain-of-thought search, GRPO-trained planners). The roadmap's hierarchical angle remains relevant because HTN's method library concept maps naturally to "skills" in agent systems — the skill selection problem in autonomous agents is structurally identical to HTN method selection. The roadmap implicitly argues that agent skill libraries should be organized as HTN method hierarchies, not flat lists.

## Related Papers

### Paper 1: Learning to Plan with Natural Language

- **Authors**: Yiduo Guo, Yaobo Liang, Chenfei Wu, Wenshan Wu, Dongyan Zhao, Nan Duan
- **Year**: 2023
- **Venue**: arXiv (Apr 2023)
- **Link**: https://arxiv.org/abs/2304.10464

**Contribution**: Proposes a training framework where LLMs learn to generate explicit task plans in natural language, then execute those plans step-by-step. A "plan verifier" model scores plan quality before execution, enabling rejection sampling of bad plans.

**Relation to main paper**: Where Puerta-Merino focuses on structured HTN methods, Guo focuses on NL plans with learned verification. Both face the same fundamental challenge: how to ensure plan correctness before execution. The NL approach is more flexible but lacks HTN's formal guarantees. The learned verifier approach could complement HTN by providing a scoring function for LLM-generated methods.

**Why it matters**: Demonstrates that learned plan verification is viable (plan verifier achieves 85%+ accuracy), suggesting a hybrid: LLM generates HTN method candidates, learned verifier scores them.

### Paper 2: iCLP — Large Language Model Reasoning with Implicit Cognition Latent Planning

- **Authors**: Sijia Chen, Di Niu
- **Year**: 2025
- **Venue**: arXiv (Dec 2025)
- **Link**: https://arxiv.org/abs/2512.24014

**Contribution**: Proposes implicit (latent) planning — instead of generating explicit textual plans, the LLM learns a latent plan representation that guides step-by-step reasoning without explicit step listing. The model learns to "plan in latent space" via a cognition planning module trained end-to-end.

**Relation to main paper**: This is the most radical alternative to HTN: no explicit plan structure at all. Where HTN says "structure the plan explicitly as a method hierarchy," iCLP says "the plan is an implicit latent variable that shapes generation without ever being decoded." This creates a three-way contrast: explicit-HTN (Puerta-Merino), explicit-NL (Guo), implicit-latent (Chen).

**Why it matters**: If latent planning works at scale, it challenges the entire premise of explicit plan generation. However, latent plans are not interpretable, debuggable, or reusable — tradeoffs the roadmap paper would identify as crucial.

## Current Understanding

Planning for LLM agents now has at least **four distinct paradigms** in our map:

1. **Pure LLM planning** (survey Day 1) — LLM generates plans via prompting; flexible but unreliable beyond ~5 steps.
2. **Hybrid LLM+classical** (Day 2) — LLM translates NL to PDDL/plan graphs; correct but domain-engineering-heavy.
3. **Hierarchical/HTN planning** (Day 3) — LLM generates or selects HTN methods; natural decomposition but domain acquisition bottleneck.
4. **Implicit/latent planning** (Day 3) — No explicit plan; latent variable guides step-by-step generation; uninterpretable but potentially more robust.

The key question is no longer "which paradigm is best" but **"how to dynamically select and combine these paradigms based on task characteristics."**

**Key Concepts Added Today:**
- Hierarchical Task Networks (HTN) — tasks decomposed via method selection, not state-space search
- LLM-HTN integration levels (0-4) — from domain authoring assistant to full LLM replacement
- Method selection vs. task decomposition — HTN's dual-level structure
- Domain acquisition bottleneck — generating structured domain models from LLMs
- Plan verification — learned verifiers that score plans before execution
- Latent planning — implicit plan representations

## Open Questions

1. HTN + LLM: Is the main bottleneck domain acquisition (generating the method library) or method selection (choosing the right method at runtime)?
2. Can learned plan verifiers (Guo) be combined with HTN method libraries to create self-correcting hierarchical planners?
3. Does latent planning (iCLP) actually outperform explicit planning on complex tasks, or is it limited to simple reasoning chains?
4. Should agent "skill libraries" be structured as HTN method hierarchies? This connects directly to the Tool Use topic's open questions about tool organization.
5. What is the evaluation methodology for LLM+HTN systems — plan feasibility, plan optimality, or execution success rate?
6. Can HTN's hierarchical structure help with the re-planning problem (how to efficiently replan when a subtask fails)?

## Possible Thesis Ideas

- **Hierarchical skill library for agents** — Structure agent skill libraries as HTN method hierarchies, with LLM-based method selection at each level. This unifies the Tool Use and Planning topics: tool selection becomes HTN method selection.
- **Self-supervised HTN domain acquisition** — Use LLMs to automatically extract HTN method hierarchies from execution traces, solving the domain acquisition bottleneck without human annotation.
- **Learned verifier for hierarchical plans** — Combine Guo-style plan verifiers with HTN structure to score method decompositions before execution, enabling efficient rejection of bad plans.
- **Adaptive planning paradigm selection** — A meta-controller that decides whether to use pure LLM planning, PDDL hybrid, HTN, or latent planning based on task complexity, available domain knowledge, and reliability requirements.
- **Plan-as-hierarchy** — Extend the plan-as-code idea (from Day 2) to hierarchical code structures: top-level plan as a function call tree, where each function is a subtask with its own decomposition.
