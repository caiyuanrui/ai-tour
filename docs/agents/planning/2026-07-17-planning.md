## Today's Question

How can hierarchical planning and context compression be combined to help LLM agents maintain coherent plans over long-horizon tasks?

## Main Paper

### Metadata

- Title: HIPIF: Hierarchical Planning and Information Folding for Long-Horizon LLM Agent Learning
- Authors: Juncheng Diao, Zhicong Lu, Peiguang Li, Yongwei Zhou, Changyuan Tian, Qingbin Li, Rongxiang Weng, Jingang Wang, Xunliang Cai
- Year: 2026
- Venue: arXiv
- Link: https://arxiv.org/abs/2606.10507

### Why this paper?

After covering the survey (Day 1), hybrid PDDL planning (Day 2), and the HTN roadmap (Day 3), a natural question is: what does a *working end-to-end system* for agent planning look like? HIPIF is a concrete, recent system that combines two recurring ideas — hierarchical task decomposition and long-context management — into a single training framework. It directly addresses the limitation we identified in the HTN roadmap: even if you decompose tasks, the growing history of observed context degrades the agent's ability to plan and track state.

### Core Problem

LLM agents in multi-turn long-horizon tasks suffer from two related problems:

1. **Long-context interference** — as the agent accumulates observations, its context window fills up, making it harder to distinguish relevant state from noise, and weakening the agent's ability to track the global task state.
2. **Sparse rewards** — in long-horizon tasks, reward signals are infrequent, making it hard for the agent to learn which actions contributed to success or failure.

Existing methods like fine-grained credit assignment (e.g., process reward models) help with (2) but not (1). Hierarchical RL methods decompose tasks to reduce long-term dependency but don't directly address the context-length problem.

### Main Idea

HIPIF proposes two complementary mechanisms:

1. **Hierarchical Planning** — the agent learns to organize execution around explicit subgoals. A high-level policy generates subgoals, and a low-level policy executes actions toward each subgoal.

2. **Information Folding** — when a subgoal is completed, the agent "folds" the completed subgoal's history into a compact summary representation. This condensed representation retains task-relevant information while discarding noisy details, preventing the context from growing unboundedly.

The agent is trained end-to-end, meaning the hierarchical decomposition, the information folding, and the action policies are all optimized jointly rather than composed separately.

### Technical Details

HIPIF uses a two-level architecture:

- **High-level planner**: takes the current state (including folded summaries of past subgoals) and produces the next subgoal. This operates at a coarse timescale — subgoals are generated after each complete sub-task, not at every step.
- **Low-level executor**: given a subgoal, produces step-by-step actions. Executes within the current subgoal until completion or failure.
- **Information folding module**: after a subgoal finishes, processes the trajectory of that subgoal (state → actions → observations) into a fixed-length compressed representation.

The key insight is that information folding is analogous to how humans handle complex tasks: we don't remember every detail of every subtask — we remember the outcome, what was learned, and what changed. HIPIF explicitly models this compression.

Training uses reinforcement learning with a combination of:
- Subgoal completion reward (did the low-level executor achieve the subgoal?)
- Task completion reward (did the overall task succeed?)
- Compression loss (encouraging the folding module to retain task-relevant information)

### Research takeaway

HIPIF's contribution is not revolutionary in any single component, but the **combination** is significant: it shows that hierarchical planning and context compression can be learned together, and that folding completed subtask histories is a viable alternative to simply truncating or paying the full context cost. This is a practical engineering solution to the planning-context tradeoff that all long-horizon agents face.

### Modern perspective

HIPIF connects to several themes from our earlier days:

- **Hierarchical planning (Day 3)**: HIPIF provides one concrete instantiation of the HTN-style decomposition — the LLM generates subgoals rather than having them pre-specified in a domain definition.
- **Information folding vs. memory (upcoming topic)**: Folding is a form of memory compression. It raises the question: should memory be explicit (stored in a retrieval database) or implicit (folded into a learned representation)? HIPIF chooses the latter, trading interpretability for end-to-end training.
- **Re-planning**: If a subgoal fails, the high-level planner can generate a new subgoal. But the paper doesn't extensively explore re-planning strategies — it's implicitly handled through the hierarchical structure.

## Related Papers

### Paper 1: Web Agents Should Adopt the Plan-Then-Execute Paradigm

- **Authors**: Julien Piet, Annabella Chow, Yiwei Hou, et al. (UC Berkeley)
- **Year**: 2026
- **Link**: https://arxiv.org/abs/2605.14290

**Contribution**: Argues that ReAct (interleaved reasoning-acting) is the wrong default for web agents. Proposes plan-then-execute: commit to a task-specific program upfront, then execute it deterministically. The motivation is security: under ReAct, untrusted web content directly steers the model's action decisions, creating a path for prompt injection. Under plan-then-execute, untrusted data can influence values inside a predefined program but cannot redefine the task.

**Relation to main paper**: Both papers challenge the ReAct-as-default assumption. HIPIF does it via hierarchical structure + information folding for long-horizon tasks; Piet et al. do it via upfront planning for security. Together they suggest a broader shift: the field is moving beyond ReAct toward architectures that separate planning from execution in various ways.

**Why it matters**: The security argument for plan-then-execute is strong and domain-independent. If web agents — a major deployment target — move to this paradigm, it changes the default architecture for thousands of deployed agents.

**Deep-read later?**: Yes — the empirical comparison between plan-then-execute and ReAct on WebArena would be valuable.

### Paper 2: A Plan Reuse Mechanism for LLM-Driven Agent

- **Authors**: Guopeng Li, Ruiqi Wu, Haisheng Tan
- **Year**: 2025
- **Link**: https://arxiv.org/abs/2512.21309

**Contribution**: Observes that ~30% of requests to LLM-driven agents are identical or similar. Proposes a plan reuse mechanism: instead of generating a new plan from scratch, the agent retrieves a semantically similar past plan and adapts it. The key challenge is defining similarity between request texts in natural language.

**Relation to main paper**: HIPIF builds plans from scratch for each task; this paper argues that many plans can be reused. Both address the efficiency problem of planning — HIPIF through hierarchical compression of subgoal histories, this paper through cross-task plan reuse. They are complementary: an ideal system might reuse past plans at the high level (subgoal structure) while using HIPIF's folding at the low level.

**Why it matters**: Plan reuse directly addresses the latency problem we noted in Day 1. Real-time agents can't afford 10+ seconds of planning per request. Experience-driven planning (reuse + adaptation) may be essential for practical deployment.

**Deep-read later?**: Yes — the similarity definition method and the adaptation mechanism are directly relevant to our "experience-driven planning" thesis idea.

## Current Understanding

Today's session adds two new dimensions to our planning map:

**Dimension 1: Hierarchical + Compression (HIPIF)**. Pure hierarchical decomposition isn't enough for long horizons — even with decomposed subtasks, the accumulated context degrades state tracking. HIPIF's answer is to fold (compress) completed subgoal histories into compact representations. This is a pragmatic design that explicitly acknowledges the context-window bottleneck, rather than assuming it away.

**Dimension 2: Plan-then-Execute vs. ReAct**. The ReAct paradigm (interleaved think-act-observe) has been the default for agent architectures. Two recent papers challenge this default from different angles: HIPIF argues it's suboptimal for long-horizon tasks (you need hierarchical structure and compression), while Piet et al. argue it's actively dangerous for web agents (creates prompt injection surface). This suggests the field is converging toward architectures that separate planning from execution more cleanly.

**Dimension 3: Plan Reuse**. Generating plans from scratch for every request is expensive and wasteful. For deployed agents, plan reuse with semantic retrieval offers a path to sub-second planning latency. The key open problem is how to define and compute plan similarity.

The four paradigms from our map now look like:

| Paradigm | Approach | Status |
|----------|----------|--------|
| Pure LLM planning | Prompt-based | Limited reliability |
| Hybrid (PDDL/graph) | LLM + classical planner | Correct but engineering-heavy |
| Hierarchical (HTN-style) | Subgoal decomposition | Practical but needs context management |
| **Hierarchical + Compression** | HIPIF-style | **Adds context management** |
| Implicit/latent | No explicit plan | Uninterpretable but robust |

## Key Concepts

- Hierarchical planning with subgoal decomposition
- Information folding (context compression for completed subtasks)
- Plan-then-execute paradigm
- Plan reuse via semantic similarity

- End-to-end training of hierarchical agent policies

## Open Questions

1. **HIPIF's folding vs. explicit memory**: Does learned compression (folding) outperform structured retrieval from a memory database? The tradeoff is capacity vs. interpretability.
2. **Re-planning under HIPIF**: How should the high-level planner react to subgoal failure — generate a new subgoal immediately, or re-examine the full plan structure?
3. **Plan reuse at scale**: Can similarity-based plan retrieval work for thousands of distinct task types, or does it only help for repetitive enterprise workflows?
4. **Plan-then-execute + HIPIF**: Can plan-then-execute be combined with HIPIF's hierarchical structure — i.e., commit to a high-level plan upfront, but use HIPIF-style folding for execution?
5. **Security of hierarchical planners**: Does the plan-then-execute security argument apply to structured planners? If the plan is generated by an LLM, can't the LLM be prompt-injected during plan generation?

## Possible Thesis Ideas

- **Unified planning + memory architecture**: Combine HIPIF's information folding with an explicit memory retrieval system — use folding for recent context compression and retrieval for historical pattern reuse. This bridges the gap between learned compression and structured memory.
- **Plan reuse with retrieval-augmented generation**: Extend the plan reuse idea with RAG-style retrieval: maintain a vector database of past plan structures (subgoal DAGs), retrieve the most similar plan for a new task, and adapt it using HIPIF's low-level executor. This could achieve sub-second planning latency with hierarchical structure.
- **Security-aware planning paradigm selection**: Build a meta-controller that dynamically chooses between ReAct, plan-then-execute, and HIPIF-style hierarchical planning based on task characteristics (security sensitivity, horizon length, action complexity).
