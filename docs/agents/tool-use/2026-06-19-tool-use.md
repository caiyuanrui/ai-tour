## Today's Question

How do language models decide when and how to use external tools?

## Main Paper

### Metadata

- **Title:** Toolformer: Language Models Can Teach Themselves to Use Tools
- **Authors:** Timo Schick, Jane Dwivedi-Yu, Roberto Dessì, Roberta Raileanu, Maria Lomeli, Luke Zettlemoyer, Nicola Cancedda, Thomas Scialom
- **Year:** 2023
- **Venue:** NeurIPS 2023
- **Link:** https://arxiv.org/abs/2302.04761

### Why this paper?

Toolformer is the canonical starting point for the tool-use topic. It was the first paper to show that language models can **self-supervise** their own tool use — no manual annotations required. Almost every subsequent tool-use method references Toolformer, making it essential for building the map.

### Core Problem

Language models are remarkably capable at few-shot learning and instruction following, yet they paradoxically fail at basic tasks that much smaller, specialized models handle trivially: arithmetic, factual lookup, time-sensitive queries, and language translation. The question is: can an LM learn to call the right tool at the right time without human supervision?

### Main Idea

Train a language model to insert special API calls into the text stream:

```
Input:  "What is 234 * 567?"
Output: "Sure, let me calculate that. [Calculator(234 * 567)] → 132678. So 234 * 567 = 132678."
```

The key insight is **self-supervised learning**: given a few hand-written demonstrations per API (as few as 10 examples), the model generates candidate API call positions, filters them by whether the API result actually reduces perplexity on the next token, and then fine-tunes on the resulting self-generated data.

### Technical Details

1. **Sampling step:** For each position in a text, the model samples whether inserting `[API(args)]` would be useful.
2. **Filtering step:** For each candidate call, execute the API and compute whether `P(next_token | context + API_result)` is lower than `P(next_token | context)`. Only keep candidates where the API result reduces perplexity.
3. **Fine-tuning step:** Fine-tune the LM on the filtered (text, API_call, API_result) sequences using standard language modeling loss.

The model learns three things simultaneously:
- When to call an API (trigger condition)
- Which API to call (selection)
- What arguments to pass (formulation)

### Research Takeaway

Toolformer showed that tool use **can emerge from self-supervision alone**, without reinforcement learning or human feedback. This was a breakthrough because it meant any LM could be taught tool use simply by having access to APIs and a handful of examples. However, the approach has significant limitations (see below).

### Modern Perspective (2026)

Toolformer's self-supervised approach was revolutionary but proved hard to scale beyond ~6 simple APIs. The field has largely moved to:
- **In-context tool use** (GPT-4 function calling, Claude tool use) — no fine-tuning needed; tools are described in the system prompt
- **Fine-tuned for API calls** (Gorilla, ToolLLM) — supervised fine-tuning on large crowdsourced API datasets
- **Reinforcement learning** (RL for tool selection) — using reward signals to improve tool use policies
- **Agentic frameworks** (LangChain, AutoGen) — tool orchestration moved to the framework level

Toolformer's core contribution — that self-supervision is sufficient for basic tool learning — remains foundational, but production systems now rely more on in-context tool descriptions and RL fine-tuning.

### Limitations

- Only demonstrated with 6 tools (calculator, QA, 2 search engines, translation, calendar)
- Self-supervision filtering is computationally expensive (must try many candidate positions)
- Cannot learn **when not to use a tool** — only learns to call tools that reduce perplexity
- Does not handle **multi-step tool composition** (chaining tools together)
- No mechanism for **tool discovery** — tools are pre-defined and fixed
- Performance on tool use degrades with API complexity (nested arguments, multiple outputs)
- Relies on the API results being correct — no error handling or retry logic

## Related Papers

### Paper 1 — Tool Learning with Large Language Models: A Survey

- **Authors:** Changle Qu, Sunhao Dai, Xiaochi Wei, Hengyi Cai, Shuaiqiang Wang, Dawei Yin, Jun Xu, Ji-Rong Wen
- **Year:** 2024
- **Venue:** Frontiers of Computer Science (arXiv:2405.17935)
- **Link:** https://arxiv.org/abs/2405.17935

**Contribution:** A comprehensive survey that organizes the tool learning field into a 4-stage workflow: (1) Task Planning — decomposing user requests into sub-tasks, (2) Tool Selection — choosing the right tool from a candidate set, (3) Tool Calling — executing the tool, (4) Response Generation — integrating tool output into the answer.

**Relation to main paper:** The survey places Toolformer within the broader taxonomy — Toolformer is a self-supervised method that primarily addresses the Tool Selection + Tool Calling stages. The survey shows how much the field has expanded beyond Toolformer's approach.

**Why it matters:** Gives the structural map of the entire tool-learning field. Essential for understanding where Toolformer fits and what alternative approaches exist (RL-based, retrieval-based, fine-tuning-based).

**Deep read later?** Yes — this survey should be used as a reference guide throughout the Tool Use topic. Each stage in the taxonomy corresponds to a subtopic worth exploring.

### Paper 2 — Gorilla: Large Language Model Connected with Massive APIs

- **Authors:** Shishir G. Patil, Tianjun Zhang, Xin Wang, Joseph E. Gonzalez
- **Year:** 2023
- **Venue:** NeurIPS 2023
- **Link:** https://arxiv.org/abs/2305.15334

**Contribution:** Gorilla fine-tunes LLaMA-7B on a large corpus of API documentation (Torch Hub, HuggingFace, TensorFlow Hub) to produce accurate API calls. Key innovation: a **retrieval-aware** training pipeline — during inference, the model sees relevant API docs retrieved from an index, enabling it to adapt to API changes without retraining.

**Relation to main paper:** Where Toolformer uses self-supervised in-context learning for 6 simple tools, Gorilla takes the **fine-tuning + retrieval** approach for thousands of complex APIs. Gorilla outperforms GPT-4 on API call accuracy, showing that fine-tuning on high-quality API data is extremely effective.

**Why it matters:** Represents the alternative paradigm to Toolformer — supervised fine-tuning with retrieval augmentation (RAG for API calls). This contrast (self-supervised vs. supervised) is a central design choice in the tool-use literature.

**Deep read later?** Yes — Gorilla's retrieval-aware training is a key technique for adapting tool use to evolving API landscapes.

## Current Understanding

Tool use in LLMs can be approached through three main paradigms:

1. **Self-supervised (Toolformer):** The LM learns to call tools by minimizing perplexity on automatically generated (API call, result) pairs. No manual annotation needed, but limited to simple tools and single-step calls.
2. **Supervised fine-tuning (Gorilla, ToolLLM):** The LM is fine-tuned on curated datasets of API call examples. Higher accuracy, especially on complex APIs, but requires expensive data collection and doesn't easily generalize to unseen tools.
3. **In-context learning (GPT-4/Claude function calling):** Tool descriptions are provided in the system prompt, and the model generates calls without any fine-tuning. Most flexible but depends on the base model's instruction-following ability and context window.

The key open question is about **decision boundaries**: when does the model decide to use a tool vs. rely on its own knowledge? Toolformer's self-supervision answers this narrowly (use tool if it reduces perplexity), but this criterion is incomplete — sometimes the model should use a tool even if the LM can guess the answer (for correctness guarantees), and sometimes it should avoid tools even if they reduce perplexity (for latency/cost reasons).

## Key Concepts

- **Self-supervised tool learning:** Learning tool use without human annotations, via perplexity-based filtering
- **API call formulation:** Generating correct arguments and formatting for tool APIs
- **Retrieval-aware tool use:** Using retrieved API documentation at inference time to handle dynamic tool catalogs
- **Tool selection vs. tool calling:** The decision of *which* tool to use is separate from the execution
- **Perplexity reduction as training signal:** Using next-token prediction improvement as a proxy for tool usefulness
- **Tool composition:** Chaining multiple tools together to solve complex tasks

## Open Questions

1. **How should the model decide between parametric knowledge and tool use?** Toolformer's perplexity-based criterion is a proxy, but the true decision boundary should consider correctness, latency, cost, and robustness.
2. **Can tool use generalize to entirely unseen APIs without fine-tuning?** In-context learning works for known API patterns but struggles with genuinely novel API schemas.
3. **What is the right taxonomy of tools?** Calculator and search are very different from code execution and database queries — should the same mechanism handle all of them?
4. **How do we evaluate tool use beyond API call accuracy?** True evaluation requires measuring whether the tool was the *right* choice, not just whether the call was syntactically correct.
5. **Can self-supervised tool learning scale to thousands of tools?** Toolformer works for 6 tools; Gorilla uses fine-tuning for scale. Is there a way to scale self-supervision?
6. **How should agents handle tool failures gracefully?** Toolformer assumes perfect API execution; real systems need error handling, retries, and fallback strategies.

## Possible Thesis Ideas

1. **Adaptive tool-use policy** — dynamically decide between calling a tool vs. relying on parametric knowledge based on a learned cost-benefit model (latency × accuracy × reliability).
2. **Self-supervised tool composition** — extend Toolformer-style self-supervision to multi-step tool chains, using subgoal decomposition to generate training data for composed tool calls.
3. **Tool-use as a hierarchical RL problem** — frame the when-to-use-tool decision as a POMDP where the action space includes both tool calls and normal text generation, and the reward includes task completion plus efficiency.
4. **Retrieval-augmented tool discovery** — combine Gorilla-style retrieval with Toolformer-style self-supervision to enable zero-shot adaptation to new tool APIs.
