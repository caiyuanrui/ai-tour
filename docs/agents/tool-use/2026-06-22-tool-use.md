## Today's Question

How can LLM agents scale from 6 hand-picked tools (Toolformer) to 16,000+ real-world APIs? What architectural patterns enable reliable tool selection and multi-step composition at scale?

## Main Paper

### Metadata

- **Title:** AnyTool: Self-Reflective, Hierarchical Agents for Large-Scale API Calls
- **Authors:** Yu Du, Fangyun Wei, Hongyang Zhang
- **Year:** 2024
- **Venue:** ICML 2024
- **Link:** https://arxiv.org/abs/2402.04253
- **Code:** https://github.com/dyabel/AnyTool

### Why this paper?

Day 1 of Tool Use introduced the core paradigms (self-supervised, SFT, ICL) with Toolformer (6 tools) and Gorilla (retrieval-aware fine-tuning). But neither paper addresses the **scaling challenge**: how do you select from 16,000+ APIs organized across 49 categories? AnyTool directly addresses this with a hierarchical retrieval architecture + self-reflection loop, outperforming ToolLLM by +35.4% on ToolBench.

### Core Problem

Given a user query and a catalog of 16,000+ real-world RESTful APIs (Rapid API Hub), the agent must:
1. **Identify** which subset of APIs are relevant to the query
2. **Plan** a multi-step sequence of API calls (when the task requires composing tools)
3. **Execute** the calls with correct parameters
4. **Self-reflect** if the solution fails — re-attempt with a revised approach

The key challenge is the **needle-in-a-haystack** problem: flat retrieval over 16K+ APIs gives low precision, and the cost/context of listing all tools is prohibitive.

### Main Idea

AnyTool introduces a **three-component architecture** with a self-reflection loop:

```
User Query
    │
    ▼
┌─────────────────────────────────┐
│ Hierarchical API Retriever      │
│  ┌─────────┐  ┌──────────────┐  │
│  │ API Tree│  │ Leaf-level   │  │
│  │ Router  │─►│ API Retriever│  │
│  └─────────┘  └──────────────┘  │
└─────────────────────────────────┘
    │ (selected API subset)
    ▼
┌─────────────────────────────────┐
│ Solver (GPT-4 function calling) │
│  ┌──── API calls ────► Result  │
│  │    (multi-step if needed)   │
│  └────────────────────────────┘ │
└─────────────────────────────────┘
    │ (solution or failure signal)
    ▼
┌─────────────────────────────────┐
│ Self-Reflection Module          │
│  Evaluates if solution is       │
│  valid. If not, re-routes       │
│  to retriever with feedback.    │
└─────────────────────────────────┘
    │ (max N iterations)
    ▼
  Final Answer
```

#### 1. Hierarchical API Retriever

The retriever uses a **two-level hierarchy**:
- **Level 1 — API Tree Router:** Given the query, the router selects the most relevant API categories (from 49 categories like "Weather", "Finance", "Maps"). This prunes the search space dramatically (e.g., from 16K → ~300 APIs in selected categories).
- **Level 2 — Leaf-level Retriever:** Within the selected categories, retrieves the specific APIs most relevant to the query. Uses embedding similarity + function-calling-aware ranking.

**Key design choice:** No external training needed — the entire retriever is powered by GPT-4's function calling ability. The "routing" is done by instructing GPT-4 to output category IDs.

#### 2. Solver

The solver takes the subset of API candidates and the original query, then uses GPT-4 function calling to construct a solution. The solver can call multiple APIs in sequence (composition), with the output of one call feeding into the next.

#### 3. Self-Reflection Module

This is the most novel component. After the solver produces a solution:
- The self-reflection module evaluates whether the solution actually satisfies the query requirements.
- If the solution is deemed "impracticable," the system **re-activates from the hierarchical retriever**, potentially selecting different APIs or categories.
- The reflection signal serves as a form of **feedback** — the retriever learns from previous failed attempts which categories/APIs were tried and whether they worked.

### Technical Details

- **API catalog:** 16,464 RESTful APIs from Rapid API Hub, covering 49 categories
- **Base model:** GPT-4 (function calling variant) — all components use the same model
- **Self-reflection:** A separate GPT-4 call evaluates whether the answer satisfies the user's request
- **Max iterations:** The self-reflection loop is limited to avoid infinite loops (typically 3 iterations)
- **AnyToolBench:** The authors introduced a revised evaluation benchmark to fix issues with prior protocols that produced artificially high pass rates

### Research Takeaway

AnyTool demonstrates three key lessons for tool-use at scale:

1. **Hierarchical selection is essential at scale.** Flat retrieval over thousands of tools gives poor precision. The two-level hierarchy (category → API) mirrors how humans navigate tool catalogs.
2. **Self-reflection is a practical substitute for reinforcement learning.** Instead of training a tool-use policy via RL, AnyTool uses a self-reflection module that evaluates solutions and re-tries. This is computationally cheaper and works with any LLM backend.
3. **The retriever is the bottleneck.** The quality of the final solution is primarily determined by whether the right APIs were retrieved. A perfect solver with wrong tools fails; a mediocre solver with the right tools often succeeds.

### Limitations

- **Reflection is expensive:** Each iteration of the self-reflection loop costs multiple GPT-4 calls
- **No learning across sessions:** Reflection signals are not accumulated into a persistent policy — each query starts fresh
- **Retrieval is brittle:** The hierarchical routing can make irreversible mistakes — if the wrong category is selected at Level 1, the right API cannot be found
- **Fixed API catalog:** Cannot dynamically discover new APIs or adapt to evolving API schemas
- **Single-threaded reasoning:** No explicit planning step before tool selection — the solver plans and calls simultaneously

### Modern Perspective (2026)

Since AnyTool (ICML 2024), the field has moved toward:
- **Tool retrieval as embedding + RL** — learned retrievers that are fine-tuned with tool-use rewards
- **Agentic tool discovery** — agents that browse API documentation dynamically (similar to how humans read docs)
- **Tool abstraction layers** — grouping related APIs into "capabilities" rather than individual endpoints
- **Unified tool representation** — representing tools as structured types (JSON Schema / OpenAPI) rather than natural language descriptions

## Related Papers

### Paper 1 — ToolLLM: Facilitating Large Language Models to Master 16000+ Real-World APIs

- **Authors:** Yujia Qin, Shihao Liang, Yining Ye, et al.
- **Year:** 2023
- **Venue:** NeurIPS 2023
- **Link:** https://arxiv.org/abs/2307.16789

**Contribution:**

1. **ToolBench dataset:** 16,464 real-world RESTful APIs from Rapid API Hub (49 categories), with automatically generated instructions covering both single-tool and multi-tool scenarios. Solution paths (chains of API calls) are annotated by ChatGPT.
2. **DFSDT (Depth-First Search-based Decision Tree):** A novel inference algorithm that enables the LLM to explore multiple reasoning traces and expand the search space during tool planning. Unlike greedy decoding (one path) or beam search (independent paths), DFSDT maintains a tree of partial solutions and backtracks when a branch fails.
3. **ToolLLaMA:** LLaMA fine-tuned on ToolBench, equipped with a neural API retriever. Shows comparable performance to ChatGPT on complex tool-use tasks and strong zero-shot generalization to unseen APIs.

**Relation to main paper:** ToolLLM is the primary baseline for AnyTool. AnyTool's +35.4% improvement on ToolBench demonstrates that self-reflection + hierarchical retrieval significantly outperforms DFSDT-style tree search alone. ToolLLM's DFSDT focuses on exploring the *execution* space (how to call tools), while AnyTool's hierarchical retriever focuses on exploring the *API selection* space (which tools to call).

**Why it matters:** ToolLLM's ToolBench dataset has become a standard benchmark for tool-use evaluation. DFSDT is a key technique for multi-step tool planning — it's the bridge between simple single-step tool use and structured planning.

**Deep read later?** Yes — DFSDT is worth understanding in detail as it connects tool use to planning algorithms.

### Paper 2 — ToolACE: Winning the Points of LLM Function Calling

- **Authors:** Weiwen Liu, Xu Huang, Xingshan Zeng, et al. (24 authors)
- **Year:** 2024 (ICLR 2025)
- **Link:** https://arxiv.org/abs/2409.00920
- **Code/data:** https://huggingface.co/Team-ACE

**Contribution:**

1. **Self-evolution synthesis:** An automatic pipeline that iteratively expands an API pool to **26,507 diverse APIs** — larger than ToolBench's 16K. The pipeline generates training data through a multi-agent interplay.
2. **Dual-layer verification:** Rule-based checks (syntax, schema compliance) + model-based checks (semantic correctness, goal fulfillment) to ensure data quality.
3. **SOTA at 8B scale:** A model trained on ToolACE data with only **8B parameters** achieves state-of-the-art performance on the Berkeley Function-Calling Leaderboard (BFC-L), rivaling the latest GPT-4 models.

**Relation to main paper:** Where AnyTool focuses on the *inference-time* architecture (hierarchical retrieval + reflection), ToolACE focuses on the *training-time* data pipeline. The two approaches are complementary: better training data (ToolACE) can improve the base model's function-calling ability, which in turn improves AnyTool's solver and retrieval components.

**Why it matters:** ToolACE demonstrates that the **data bottleneck** — not the model size or architecture — is the primary constraint on function-calling ability. With sufficiently good synthetic training data, small models can match GPT-4-class performance. This has major implications for cost and deployability.

**Deep read later?** Yes — the self-evolution synthesis pipeline is a key technique for generating training data without human annotation, applicable beyond tool use to many LLM capabilities.

## Current Understanding

The tool-use landscape now has a more structured map:

### Three Paradigms (from Day 1)
| Paradigm | Method | Scale | Training Cost | Inference Cost |
|----------|--------|-------|---------------|----------------|
| Self-supervised | Toolformer | ~6 tools | Medium | Low |
| SFT + Retrieval | Gorilla / ToolLLaMA | 1K–16K tools | High | Medium |
| ICL + Function Calling | GPT-4 / AnyTool | 16K+ tools | None | High |

### Two Scaling Challenges

**1. Selection at scale (AnyTool contribution)**
- Flat retrieval doesn't work at 16K+ tools
- Hierarchical routing (category → API) is the current best practice
- Self-reflection acts as a quality gate for tool selection

**2. Multi-step composition (ToolLLM contribution)**
- Single tool calls cover only ~30% of real-world queries
- DFSDT explores multiple execution paths at inference time
- The composition problem is related to **planning** (the next topic in the agents course)

### The Data Bottleneck (ToolACE contribution)
- Function calling is primarily a **data quality** problem, not a model architecture problem
- Synthetic data pipelines (multi-agent, self-evolution) can produce high-quality training data at scale
- 8B models with good data rival GPT-4 on function calling

## Key Concepts

- **Hierarchical API retrieval:** Two-level (category → API) selection to handle 10K+ tools
- **Tool composition:** Multi-step API chains where one API's output feeds another's input
- **DFSDT:** Depth-first search decision tree for exploring multiple tool execution paths
- **Self-evolution synthesis:** Iterative pipeline that grows both API coverage and training data quality
- **Self-reflection loop:** Evaluating solution quality and re-attempting with feedback
- **Dual-layer verification:** Rule-based + model-based checks on tool-use training data
- **Function calling at scale:** Using LLM function calling APIs as a general-purpose tool-use interface

## Open Questions

1. **Hierarchical vs. flat retrieval:** Is the 2-level hierarchy optimal, or do we need deeper (3–4 level) hierarchies for specialized domains?
2. **Reflection vs. RL:** Can self-reflection at inference time match RL-trained policies, or is RL training essential for reliable tool use?
3. **Planning vs. reactive tool use:** When should tool use be planned (structured plan before execution) vs. reactive (ReAct-style interleaved call/observe)?
4. **Tool abstraction:** Should we expose individual APIs or higher-level "capabilities" (e.g., "search the web" vs. specific search APIs)?
5. **Dynamic tool discovery:** How do agents discover and incorporate new APIs that weren't in their training catalog?
6. **Evaluation beyond BFC-L:** Berkeley Function-Calling Leaderboard measures call correctness, but what about call *necessity* (was the tool actually needed)?
7. **Cost-quality tradeoff:** The best methods (AnyTool + GPT-4) are the most expensive. Can small models + good data (ToolACE) close the gap?

## Possible Thesis Ideas

1. **Adaptive retrieval hierarchy** — dynamically construct the retrieval tree based on query complexity rather than using a fixed hierarchy, balancing breadth vs. depth based on available context window.

2. **Tool-use data flywheel** — combine AnyTool's self-reflection with ToolACE's self-evolution: use reflection failures as signal to generate hard negative training examples, improving the model iteratively.

3. **Unified planning-tool-use architecture** — extend DFSDT into a hierarchical planner where high-level plans select capabilities and low-level plans select specific APIs, connecting tool use to the planning topic.

4. **Cost-aware tool selection policy** — learn to trade off between calling a tool (cost, latency) vs. using parametric knowledge, with a learned cost model for each API endpoint.
