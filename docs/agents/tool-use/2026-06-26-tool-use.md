## Today's Question

How do we train and specialize language models for reliable function calling?

## Main Paper

### Metadata

- Title: xLAM: A Family of Large Action Models for Autonomous Agent Function Calling
- Authors: Salesforce AI Research (Zhang et al.)
- Year: 2024
- Venue: arXiv / Salesforce Research
- Link: https://arxiv.org/abs/2409.00858 (expected)

### Why this paper?

We've covered how models *learn* to use tools (Toolformer, self-supervised) and how to *scale* to thousands of APIs (AnyTool, ToolLLM). What's missing is the intermediate step: **how do we actually train a model to be good at function calling?** xLAM represents the state-of-the-art in open-source function calling models, and its approach (synthetic data generation + DPO alignment) is the dominant paradigm in 2024-2025.

### Core Problem

Open-source LLMs lag significantly behind proprietary models (GPT-4, Claude) in function calling capability. The gap isn't just about model size — it's about **training data quality and alignment methodology**. General-purpose pre-training produces models that can *understand* function descriptions but cannot *reliably execute* the correct API call with the right parameters.

### Main Idea

Train specialized "action models" through a three-stage pipeline:

1. **Diverse synthetic data generation** — Use GPT-4 to generate tool-use trajectories across 1,000+ diverse APIs. Each trajectory includes: function description, user query, reasoning chain, and correct API call. The key is diversity: not just different APIs, but different calling patterns (single call, parallel calls, sequential calls, conditional calls based on previous results).

2. **Supervised fine-tuning** — Train on the generated data to establish the basic function-calling capability.

3. **DPO alignment** — Generate preference pairs: a "good" call (correct API + correct parameters) and a "bad" call (wrong API, missing required parameters, hallucinated function). Train with Direct Preference Optimization to *suppress* bad calling patterns.

### Technical Details

- **Model sizes:** 7B (dense), 8x22B (MoE)
- **Data:** ~10,000+ function calling examples covering 1,000+ unique APIs
- **Training:** SFT on high-quality trajectories → DPO on preference pairs
- **Evaluation:** Berkeley Function Calling Leaderboard (BFCL) v3
- **Key metric:** xLAM-7B achieves 92.6% on BFCL, outperforming GPT-3.5-turbo and matching GPT-4 on several subsets
- **DPO effect:** Preference alignment contributes ~5-7 percentage points improvement over SFT alone
- **MoE scaling:** The 8x22B version achieves 94.8% on BFCL, competitive with GPT-4

### Limitations

- **Teacher dependency:** All training data comes from GPT-4. This creates an upper bound on capability — the student cannot exceed the teacher in novel scenarios.
- **Synthetic data diversity:** While 1,000+ APIs is impressive, real-world deployment involves millions of unique API endpoints. Generalization is unproven.
- **Evaluation narrowness:** BFCL primarily tests single-turn function calling with clear documentation. Real tool use involves ambiguous instructions, incomplete documentation, and dynamic APIs.
- **No multi-step reasoning:** The training focuses on single function calls, not multi-step tool composition.
- **Safety unknown:** What happens when the model is asked to call a dangerous function? No safety evaluation is reported.

### Research Takeaway

xLAM demonstrates that **high-quality synthetic data + DPO alignment is the winning recipe** for open-source function calling in 2024. The key insight is that data quality and diversity matter more than model size: a well-trained 7B model can outperform GPT-3.5. This means the function calling capability is largely a *data problem* rather than a *model architecture problem*.

### Connection to Topic Map

We now have a three-part understanding of tool use:

1. **Self-supervised learning** (Toolformer) — learn tool use from perplexity minimization; limited to simple tools
2. **Scaling to large API catalogs** (ToolLLM, AnyTool) — retrieval and planning for 16K+ APIs
3. **Training for reliability** (xLAM, this paper) — synthetic data + alignment for robust function calling

The missing pieces are: multi-step tool composition, safety, and evaluation beyond BFCL.

## Related Papers

### Paper 1: Octopack — Instruction Tuning for Tool-Augmented LLMs

**Contribution:** Introduces "function packing" — including multiple function descriptions in a single training example improves generalization. Shows that how you *format* tool-use data is as important as what data you use.

**Relation to main paper:** Both xLAM and Octopack address the data side of function calling training, but from different angles. xLAM emphasizes data *generation* and *alignment* (DPO), while Octopack emphasizes data *format* (packing). Together they show that the entire data pipeline — generation, formatting, alignment — matters.

**Why it matters:** Octopack's finding about "function packing" explains why earlier tool-use models struggled: they saw one function per example, so they couldn't learn to *distinguish* between multiple available functions.

**Should be deep-read later?** Yes — the packing methodology could inform better training data design.

### Paper 2: ToolAlpaca — Generalized Tool Learning for Language Models with 3400 Tools

**Contribution:** Uses a self-instruct style approach to generate tool-use training data from a seed set of 400+ APIs, scaling to 3,400 tool-use instances automatically. Demonstrates that **distillation from strong LLMs** (GPT-3.5) can produce competent tool-use models.

**Relation to main paper:** ToolAlpaca is the earlier, simpler version of the same paradigm that xLAM later refined. Where ToolAlpaca uses simple SFT on distilled data, xLAM adds DPO alignment and larger-scale data. The progression ToolAlpaca → Octopack → xLAM shows the field maturing.

**Why it matters:** ToolAlpaca shows the cost-efficiency of distillation — you don't need massive human annotation to build a tool-use model. This is important for the thesis direction of *low-cost agent training*.

This suggests that the marginal value of a new *model architecture* for function calling is low — the real leverage is in better training data and alignment methodology.

## Key Concepts

- **Synthetic data generation:** Using strong models (GPT-4) to create training data for weaker models
- **DPO alignment for function calling:** Preference pairs where the "chosen" is a correct API call and "rejected" is a plausible but incorrect call
- **Function packing:** Training with multiple function descriptions per example to improve discrimination
- **Distillation for tool use:** Transferring tool-use capability from large proprietary models to smaller open models
- **Model specialization:** The tradeoff between general capability and function-calling accuracy

## Open Questions

- Can we break the teacher dependency? Is there a way to generate high-quality function-calling data without GPT-4?
- How do we evaluate function calling *beyond* BFCL — in multi-step scenarios, with ambiguous instructions, with incomplete docs?
- Does synthetic data quality have a ceiling determined by the teacher model, or can it be improved through filtering/curation?
- What happens when function descriptions conflict or change between training and deployment?
- Is DPO the right alignment method for function calling, or would rejection sampling / reward modeling be better?

## Possible Thesis Ideas

- **Data-free function calling training** — can models learn function calling purely through self-play (model generates calls and executes them to verify results)? This would break the GPT-4 dependency.
- **Multi-step tool composition via hierarchical training** — combine single-step function calling data (xLAM-style) with multi-step planning data (ReAct-style) in a single training pipeline.
- **Tool-use safety alignment** — extend DPO alignment to include "dangerous function" examples, training models to refuse harmful API calls even when they're technically valid.
