## Today's Question

**When an agent composes multiple tool calls, how do we ensure the composition itself is safe, not just each individual call?**

## Main Paper

### Metadata

- Title: ChainCaps: Composition-Safe Tool-Using Agents via Monotonic Capability Attenuation
- Authors: Xiaochong Jiang, Shiqi Yang, Ziwei Li, Lifei Liu, Haoran Yu, Yichen Liu
- Year: 2026
- Venue: ICML 2026 Workshop (AIWILD: Agents in the Wild)
- Link: https://arxiv.org/abs/2605.26542

### Why this paper?

Previous work on tool use focused on correctness (does it call the right API?) and training (can we make models reliable at calling tools?). But a critical gap remains: what happens when an agent chains multiple tools? The composite effect can be unsafe even when every per-tool permission check passes. This is a new dimension — **tool composition safety** — that hasn't been explored in our previous readings.

### Core Problem

**Permission laundering**: A tool-using agent can satisfy every per-tool permission check and still produce an unsafe end-to-end effect. Example: reading a confidential document (permitted for tool A), summarizing it (tool B), and sending the summary to an external endpoint (tool C). Each individual operation is permitted, but the chain leaks data.

### Main Idea

**ChainCaps** introduces a runtime capability model with a key rule: monotonic capability attenuation. Every value carries a sink-specific capability budget. When tools are composed, the capability budget propagates by **intersection** — a value can lose authority as it moves through the chain, but it can never GAIN authority through composition.

This is implemented as a transparent **MCP (Model Context Protocol) proxy** that sits between the agent and tool servers, intercepting tool calls and tracking capability budgets. No changes required to the agent or tool implementations.

### Technical Details

- **Capability budget**: each value tagged with per-sink permissions (e.g., "read file X", "write to network endpoint Y")
- **Intersection propagation**: when tool B receives output from tool A, the result carries `budget(B) ∩ budget(A)` — the most restrictive of both
- **Implementation**: MCP proxy intercepts all tool calls at the protocol level, tracks data dependencies, enforces attenuation
- **Key finding**: manifest quality is the dominant deployment bottleneck — expert manifests achieve 100% attack blocking, naive manifests fall to 27.3%

### Research takeaway

Tool composition safety is a genuinely new problem that existing safety approaches (per-tool permissions, RLHF safety alignment, prompt-level controls) don't address. The core insight — capabilities should monotonically attenuate through composition — is elegant and principled. The MCP proxy implementation is pragmatic for real-world deployment.

### Modern perspective

This paper, along with ChainFuzzer and ToolPrivBench (see related papers), signals an emerging subfield: **agent tool use safety**. The pattern is clear: as agents become more capable with tools, composition-level safety becomes the bottleneck. These three papers, all from 2026, suggest this will be a hot research direction.

## Related Papers

### Paper 1

- **Title:** When Lower Privileges Suffice: Investigating Over-Privileged Tool Selection in LLM Agents
- **Authors:** Kaiyue Yang et al.
- **Link:** https://arxiv.org/abs/2606.20023
- **Relevance:** Addresses the related problem of privilege escalation in tool SELECTION (not just composition)
- **Key points:**
  - Introduces ToolPrivBench to evaluate over-privileged tool selection
  - Finds that LLM agents commonly select higher-privilege tools despite sufficient lower-privilege alternatives
  - Transient failures amplify this tendency (agents escalate on first failure)
  - General safety alignment doesn't reliably transfer to least-privilege tool choice
  - Proposes privilege-aware post-training defense that substantially reduces unnecessary high-privilege tool use

### Paper 2

- **Title:** ChainFuzzer: Greybox Fuzzing for Workflow-Level Multi-Tool Vulnerabilities in LLM Agents
- **Authors:** Jiangrong Wu et al.
- **Link:** https://arxiv.org/abs/2603.12614
- **Relevance:** Provides a testing/verification methodology for multi-tool vulnerabilities
- **Key points:**
  - Studies multi-tool vulnerabilities where data flows through tool chains (source-to-sink)
  - Proposes ChainFuzzer: extracts tool chains, synthesizes stable prompts, fuzzes with guardrail awareness
  - Evaluated on 20 open-source LLM agent apps (998 tools), found 365 unique vulnerabilities
  - 302 of 365 require multi-tool execution (single-tool testing misses them)
  - Trace-guided Prompt Solving increases chain reachability from 27% to 95%

## Current Understanding

Tool use research has expanded beyond "can the model call the right tool?" to a more nuanced set of problems:

1. **Self-supervised tool learning** (Toolformer) — models teach themselves via perplexity reduction
2. **Hierarchical retrieval + reflection** (AnyTool) — scaling to large tool catalogs
3. **Training for reliability** (xLAM, ToolACE) — synthetic data + alignment for function calling
4. **Tool composition safety** (ChainCaps, ToolPrivBench, ChainFuzzer) — new dimension: safety of multi-step tool chains

The emerging picture: as tools become abundant and agents chain them freely, **composition-level safety** becomes the limiting factor. Per-tool permission checks, prompt-level controls, and safety alignment all fail to address permission laundering and privilege escalation. The field is converging on capability-based approaches (monotonic attenuation, least-privilege selection) as principled solutions.

## Key Concepts

- Permission laundering: unsafe end-to-end effect from individually-permitted tool calls
- Monotonic capability attenuation: value cannot gain authority through tool composition
- Intersection propagation: composed tool output inherits the most restrictive budget
- MCP proxy: transparent protocol-level enforcement without agent/tool modification
- Over-privileged tool selection: agent choosing higher-privilege tools than needed
- Least-privilege tool selection: prefer sufficient, not maximal, tool capabilities
- ToolPrivBench: benchmark for measuring over-privileged selection
- Multi-tool vulnerability: security issue that only emerges through tool composition
- Source-to-sink dataflow: data from one tool becomes input to another, enabling exploitation
- Guardrail-aware fuzzing: testing under realistic LLM safety guardrails

## Open Questions

- Can monotonic capability attenuation be bypassed through side channels (e.g., timing, encoding, error messages)?
- How do we define "capability budgets" for general-purpose tools like code interpreters?
- Does privilege-aware post-training generalize to entirely unseen tool categories?
- What is the right granularity for tool manifests — should every API parameter have a capability?
- Can tool composition safety be solved at the agent architecture level (not just proxy level)?
- Is there a fundamental tradeoff between agent autonomy and composition safety?
- How should capability budgets interact with agent memory — should recalled memories inherit the budget of the agent that created them?

## Possible Thesis Ideas

- **Unified tool safety framework** — combine monotonic capability attenuation (ChainCaps) with least-privilege selection training (ToolPrivBench) and automated vulnerability discovery (ChainFuzzer) into a single provably-safe agent tool-use architecture
- **Capability-aware agent memory** — extend capability budgets to agent memory, preventing information from escaping its original permission context through memory recall and re-use
- **Adaptive tool privilege scaling** — dynamically determine the minimum required privilege for a task by learning from successful completion trajectories, then train agents to prefer that privilege level
- **Safety-composability verification for agent workflows** — design a formal verification framework that can prove end-to-end safety of multi-tool agent workflows
- **Tool manifests as learned specifications** — automatically generate capability manifests from tool documentation and example usage, reducing the expert manifest bottleneck
