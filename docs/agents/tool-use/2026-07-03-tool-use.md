## Unified Understanding

Tool use in LLMs has a clear progression:

```
Phase 1 (Self-supervised): Toolformer → model learns to call APIs via perplexity reduction
Phase 2 (Supervised): Gorilla, ToolLLM → curated datasets + retrieval for many APIs
Phase 3 (Alignment): xLAM, ToolACE → synthetic data + DPO for reliable function calling
Phase 4 (Safety): ChainCaps → capability budgets + monotonic attenuation
```

The trajectory is: **can it call tools? → can it call many tools? → can it call tools reliably? → can it call tools safely?**

## Key Concepts (consolidated)

- Self-supervised tool learning (perplexity reduction signal)
- Supervised fine-tuning with API documentation (retrieval-aware)
- Synthetic data generation + DPO alignment (open-source competitive with GPT-3.5)
- Hierarchical API retrieval (two-level: capability search → API selection)
- Self-reflection + DFS-based decision tree for multi-step tool use
- Monotonic capability attenuation for composition safety
- Least-privilege tool selection as a training objective
- Permission laundering: capability escape through composition

## Open Questions (most important)

- **Can tool use generalize to entirely unseen APIs without fine-tuning?**
- **How do we evaluate multi-step tool composition, not just single API call accuracy?**
- **Is DPO the right alignment method, or would rejection sampling / reward modeling be better?**
