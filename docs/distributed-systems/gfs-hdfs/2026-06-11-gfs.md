## Today's Question

How does Google File System (GFS) achieve high throughput and fault tolerance on commodity hardware?

## Core Paper

**"The Google File System"**

- **Authors:** Sanjay Ghemawat, Howard Gobioff, Shun-Tak Leung
- **Year:** 2003
- **Venue:** SOSP 2003
- **Link:** https://research.google/pubs/pub51/

### Core Problem

Build a distributed file system for Google's web crawl and search index pipelines. Requirements:
- Commodity hardware (failures are the norm, not the exception)
- Multi-GB files (hundreds of TB total)
- Large streaming reads (sequential)
- Append-heavy writes, rare random overwrites
- High concurrency from many clients

### System Architecture

**Single master + multiple chunkservers**:

```
Client → Master (metadata)
    ↘ Chunkserver 1 (data chunks)
    ↘ Chunkserver 2
    ↘ Chunkserver N
```

**Key design decisions:**

1. **Chunk size 64MB** — much larger than typical FS blocks (4-16KB). Reduces metadata size, enables client-side caching of chunk locations, amortizes overhead of large sequential reads.

2. **Single master** — holds all metadata in memory (only ~64 bytes per chunk). Simplifies coordination. The master is NOT a bottleneck because data flows directly between clients and chunkservers for reads/writes.

3. **Lease-based mutation** — for writes, the master grants a lease to one replica (primary), which defines the mutation order. Other replicas follow the primary. This avoids expensive consensus on every write.

4. **Shadow masters** — read-only replicas of the master for hot standby.

### Read Path

1. Client sends filename + chunk index to master
2. Master returns chunk handle + replica locations (cached at client)
3. Client contacts nearest chunkserver directly
4. Data flows: chunkserver → client (no master involvement)

**Key optimization:** the 64MB chunk size means the client fetches chunk locations once and reuses them for many reads.

### Write Path

1. Client pushes data to all replicas (nearest first, pipelined)
2. Client sends write request to primary (lease holder)
3. Primary assigns sequence numbers, applies mutation locally
4. Primary forwards serialized mutation to secondary replicas
5. Secondaries reply; primary replies to client

**If a replica fails:** the primary retries. If the lease expires without master contact, the master grants a new lease to another replica.

### Fault Tolerance

| Failure | Mechanism |
|---------|-----------|
| Chunkserver crash | Re-replicate from remaining replicas (3x default) |
| Master crash | Shadow master takes over; operation log replayed |
| Data corruption | Checksum on every 64KB block within each chunk |
| Disk failure | Chunk replication handles it; re-replication triggered |

Each chunk is replicated across **3 chunkservers** by default. Re-replication priority based on how far below 3 the count has fallen.

### Research Takeaway

GFS's key insight: **for large, append-heavy workloads, relaxing strict POSIX semantics and optimizing for sequential access is the right tradeoff**. Specifically:

- Append-only mutations (no random writes) → simplifies consistency model
- 64MB chunk → reduces metadata overhead to ~0.01% of storage
- Lease-based mutation → single-decider without distributed consensus per write

### Modern Perspective

GFS's architecture directly shaped HDFS. Key things that aged well:
- **Single master** now has HA via ZooKeeper-based failover (HDFS HA)
- **64-128MB block size** is still standard in HDFS
- **Lease-based writes** → HDFS pipeline writes

Things that didn't:
- **Append-only** is too restrictive for modern workloads; HDFS added `hflush/hsync` for read-after-write consistency
- **Single master** is a bottleneck for metadata-heavy workloads; solved via HDFS Federation + observer NameNodes
- **Shadow masters** replaced by dual-standby HA with shared edits (QJM)

## Key Concepts

- Single master + multiple chunkservers
- 64MB chunk size (large block for sequential throughput)
- Lease-based mutation ordering
- Data flow separate from control flow
- 3x replication for fault tolerance
- Checksum per 64KB block within chunk
- Append-heavy workload assumption

## Open Questions

- Why 64MB specifically? Could larger blocks (256MB, 1GB) be better for modern hardware?
- What happens when the master's metadata exceeds available memory? (HDFS solves this with Federation)
- How does the lease mechanism interact with network partitions?
