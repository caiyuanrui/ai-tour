## Today's Question

## Main Paper

### Metadata

- Title: Make Each Token Count: Towards Improving Long-Context Performance with KV Cache Eviction
- Authors: Ngoc Bui, Hieu Trung Nguyen, Arman Cohan, Rex Ying
- Year: 2026
- Venue: arXiv
- Link: https://arxiv.org/abs/2605.09649

### Why this paper?

Day 1 established that compression (quantization) reduces the per-entry bit-width of KV cache entries. The natural next question is: can we simply delete some entries entirely? This paper tackles that question directly, and critically — it does not assume eviction always degrades performance. The key insight is that *full-cache attention is not always necessary*.

### Core Problem

Existing KV eviction methods reduce memory but typically degrade accuracy relative to full-cache inference. The assumption has been: eviction is a necessary evil — you trade quality for memory. This paper challenges that assumption.

### Main Idea

The paper argues that standard eviction policies (keep tokens with highest accumulated attention scores) are suboptimal for two reasons:

1. **Attention redistribution**: When a token is evicted, adjacent tokens' attention weights shift to absorb its role. The distribution of attention *after* eviction is not the same as before — but current eviction policies plan as if it were.
2. **Task-dependent importance**: A token that is unimportant for one layer's computation may be essential for another. Per-token importance is not a scalar; it's layer-dependent.

The proposed method learns a **importance-aware eviction policy** that accounts for both attention redistribution effects and layer-specific token importance. The method uses a lightweight predictor trained to estimate the impact of evicting each token, conditioned on the remaining cache state.

### Technical Details

- **Importance scoring**: Each token gets a per-layer importance score based on its contribution to output representations. Tokens with low importance across all layers are candidates for eviction.
- **Eviction-aware retraining**: Instead of training the model on full-cache attention then evicting at inference, the method fine-tunes with actual eviction patterns so the model learns to work with the sparse cache.
- **Adaptive budget**: The eviction rate is not fixed — it adjusts based on sequence length and task difficulty, determined by a lightweight gating network.

### Research takeaway

This paper reframes the KV cache eviction problem: instead of asking "how do we pick which tokens to keep?", it asks "how do we make the model robust to missing tokens?" This is an important shift from a selection-centric view to a training-centric view.

### Modern perspective

The paper is from Rex Ying's group and is very recent (May 2026). It connects to the broader theme of **learned cache policies** — replacing hand-crafted heuristics (accumulated attention, recency) with learned predictors. This is analogous to the broader ML-for-systems trend (learned index structures, learned schedulers).

## Related Papers

### Paper 1: EVICPRESS — Joint KV-Cache Compression and Eviction

- **Authors**: Shaoting Feng, Yuhan Liu, et al.
- **Year**: 2025
- **Venue**: arXiv
- **Link**: https://arxiv.org/abs/2512.14946

**Contribution**: Proposes a unified framework that jointly optimizes compression AND eviction for KV cache, rather than treating them as separate stages. The key insight is that decision boundaries affect each other — aggressive compression changes which tokens become important, and eviction changes the set of tokens available for compression.

**Relation to main paper**: EVICPRESS is system-level optimization (how to combine two mechanisms efficiently), while Make Each Token Count is representation-level (how to make the model robust to missing tokens). They are complementary: EVICPRESS addresses the systems challenge of combining eviction + compression, while Make Each Token Count addresses the accuracy challenge of eviction alone.

**Worth deep-reading later?**: Yes — jointly optimizing compression and eviction directly addresses an open question from Day 1.

### Paper 2: Efficient Streaming Language Models with Attention Sinks (StreamingLLM)

- **Authors**: Guangxuan Xiao, Yuandong Tian, Beidi Chen, Song Han, Mike Lewis
- **Year**: 2023
- **Venue**: arXiv (later ICLR 2024)
- **Link**: https://arxiv.org/abs/2309.17453

**Contribution**: Introduced the concept of **attention sinks** — the observation that initial tokens (especially the BOS token) receive disproportionate attention simply because they are "seen" by all subsequent tokens during training. StreamingLLM's key technical finding: if you keep just the first 4 initial tokens (attention sinks) plus a sliding window of recent tokens, the model maintains stable perplexity for streaming text up to 4M+ tokens without ever needing the full cache.

**Relation to main paper**: StreamingLLM is the canonical foundation for KV cache eviction. It established two fundamental eviction principles: (1) always preserve attention sinks, (2) recent tokens matter most. Make Each Token Count builds on this foundation by challenging the implicit assumption that accumulated attention is a sufficient importance signal.

**Worth deep-reading later?**: Already canonical. Read the full paper if you want the exact experimental setup and the elegant analysis of why attention sinks form (Softmax's non-negativity forces probability mass onto the first token, which has no prior tokens to absorb it).

## Current Understanding

The KV cache management landscape now has three pillars:

1. **Quantization** (Day 1: KIVI, KVSink) — reduce bits per entry
2. **Eviction** (Day 2: Make Each Token Count, StreamingLLM) — remove entries entirely
3. **Offloading** — move entries to slower tiers (CPU, disk)

Key insight from Day 2: eviction is not a monolithic problem. There are three important sub-problems:

- **What to evict?** — importance scoring (StreamingLLM: attention sinks + window; Make Each Token Count: learned importance accounting for attention redistribution)
- **How to make models robust to eviction?** — Train with eviction patterns (Make Each Token Count's approach)
- **How to combine eviction with compression?** — EVICPRESS's insight that these interact non-trivially

The relationship between Day 1's quantization and Day 2's eviction is becoming clearer: quantization compresses what you keep, eviction decides what to keep. They operate on different dimensions, but their interaction (EVICPRESS) opens new optimization space.

## Key Concepts

- **Attention sink**: First few tokens that absorb disproportionate attention; must always be preserved during eviction
- **Sliding window + sink**: StreamingLLM's dual-component cache: first 4 tokens + last N tokens
- **Attention redistribution**: Evicting a token changes the attention distribution of remaining tokens — effective eviction must account for this
- **Eviction-aware training**: Fine-tuning the model with actual eviction patterns so it adapts to sparse KV cache
- **Joint compression-eviction**: Systems that simultaneously decide which entries to compress and which to evict

- For agent workloads (short tool-interleaved sequences), is eviction even needed? The sequences are short but accumulate across turns — does streaming eviction work differently here?
- Can EVICPRESS-style joint optimization be extended to three-way (quantization + eviction + offloading)?
- How does the importance-aware eviction policy from Make Each Token Count perform on reasoning tasks where earlier tokens (e.g., intermediate CoT deductions) become critical?

## Possible Thesis Ideas

- **Three-way unified KV cache manager**: A learned policy that jointly decides: which tokens to quantize (at what bit-width), which to evict entirely, and which to offload to CPU — with the optimization objective being SLO-constrained goodput under a memory budget. This would bring together KIVI (quantization), Make Each Token Count (eviction), and KVSwap (offloading).
- **Agent-aware KV cache scheduling**: For multi-turn agents, design an eviction policy that reuses KV cache across turns, recognizing that certain tokens (system prompts, tool definitions, previous conversation context) are re-accessed repeatedly. This extends StreamingLLM's window+sink to agent-specific patterns.
- **Attention-sink-free architecture exploration**: Investigate whether the attention sink phenomenon is a necessary consequence of Softmax attention or an artifact that can be designed away (e.g., via normalization changes). If sinks can be eliminated, eviction policies become simpler.
